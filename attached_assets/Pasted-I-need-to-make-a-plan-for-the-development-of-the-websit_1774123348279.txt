High‑level checklist (quick)
- Canonical model: normalized gifts, gift_requirements, gift_types, gift_rules, gift_sections, gift_triggers.

- Deprecate legacy columns: requires_1..19, type_1..8 → backfill then remove.
- Migration pattern: expand → migrate → contract with dual‑write and feature flags.

Recommended schema & structural changes
Core changes
- Use surrogate PKs (BIGINT AUTO_INCREMENT) and FK constraints for integrity.
- Move repeated inline columns into rows: requires_* → gift_requirements(req_id, gift_id, kind, ref_id, qty, slot).
- Normalize types into gift_types + gift_type_map (many‑to‑many).
- Store parsed @@SECTION/@@ACTION in gifts.effect_json JSON and also extract searchable fields into virtual/generated columns for indexing. Use MariaDB JSON validation.
Naming, constraints, and indexes
- Naming: lowercase, snake_case, singular table names.
- Constraints: NOT NULL where possible; CHECK for enums; UNIQUE on (slug, book_id).
- Indexes: index gift_id, type_id, kind, and JSON-extracted keys via generated columns.

Operational & platform recommendations
- Zero‑downtime migration: follow expand/migrate/contract; add columns/tables first, backfill in batches, enable dual‑write, then switch reads.
- Backfill tooling: write idempotent ETL (Python) with logging and a manual review queue for ambiguous parses.
- Testing: unit tests for parser, staging run with production snapshot, and data‑consistency checks.
- Monitoring & audit: add created_by, updated_by, created_at, updated_at, and an audit_log table for changes to gifts.
- Performance: consider partitioning very large tables (time or book_id) and caching read-heavy endpoints in the app layer.
- WordPress integration: follow CustomTables plugin patterns and avoid bypassing WP DB APIs for migrations; keep plugin upgrade scripts idempotent.

Trade-offs and risks (detailed)
- Parsing risk: free‑text @@SECTION parsing will produce edge cases—log and surface ambiguous rows for manual fix.
- JSON vs columns: JSON preserves flexibility but is less queryable; mitigate by extracting hot keys to generated columns.
- Operational risk: long backfills can impact I/O—use batched jobs and run during low traffic windows.

Species

- Add FK constraints and seed lookup tables for common trait names. Refactoring databases safely is recommended.
Equipment and Weapons
- Split equipment into base equipment and subtype tables (weapons, armor) to avoid many nullable columns.

Trappings

Careers
